Standard Operating Procedure

IT & Security Requirements for KU Automation Demo Applications

Effective Date: 08 May 2026
Review Date: 08 August 2026
Owner: Security & IT Operations

1. Purpose & Scope

This Standard Operating Procedure defines the technical infrastructure requirements and security controls for all deployment options of the KU Automation demo application suite. It ensures consistent, secure deployments that protect both KU Automation and client environments from security breaches, data leakage, and operational risks.

Applies to: Landing pages, ROI calculators, one-pagers, API mocks, document processing demos, and any internal admin interfaces provided to clients for evaluation purposes.

2. Audience

  • Developers / DevOps Engineers — Deploying and maintaining demo environments
  • Security / IT Operations Teams — Reviewing, approving, and auditing deployments
  • Client-Facing Consultants — Presenting demos and communicating requirements to clients
  • Client IT Teams — Evaluating integration requirements and security posture

3. Definitions

Term Definition
Demo App Any KU Automation showcase application provided to clients for evaluation (landing page, ROI calculator, one-pager, API mock, admin UI, document processor).
Deployment Option The hosting method used to run the demo (static hosting, Docker container, Kubernetes, on-premises VM, or local development server).
Security Controls Technical and procedural safeguards including network isolation, identity management, data protection, and logging.
Client System Any hardware, software, or network infrastructure owned or managed by the client that interacts with the demo application.
PII Personally Identifiable Information — any data that could identify an individual.

4. Deployment Options & IT Requirements

4.1 Option A: Static Web Hosting (Vercel / Netlify / Cloudflare Pages)

Attribute Specification
Components HTML/CSS/JS front-end (landing page, ROI calculator, one-pager); optional analytics scripts
Infrastructure Git repository (GitHub/GitLab/Bitbucket); CI/CD pipeline via platform integration
Runtime Platform-managed (Node.js 14+ for build process)
Configuration vercel.json or netlify.toml — defines redirects, security headers, cache policies
Use Case Client-facing demos with no backend processing; rapid deployment; global CDN distribution

4.2 Option B: Docker Container (Local, Cloud VM, or Container Service)

Attribute Specification
Components Node.js/Express API mock; Python Flask micro-service (PDF parser, data transformer); optional Nginx reverse proxy
Infrastructure Docker Engine 20.10+; minimum 2 vCPU, 4 GB RAM; 20 GB storage for images and logs
Runtime Linux (Ubuntu 22.04 LTS or Debian 12 recommended)
Configuration Dockerfile, docker-compose.yml, .dockerignore
Use Case Demos requiring backend processing; isolated client environments; reproducible deployments

4.3 Option C: Kubernetes (EKS / GKE / AKS / On-Premises)

Attribute Specification
Components Containerised services (as per Option B); Ingress controller (NGINX/Traefik); ConfigMaps & Secrets
Infrastructure 3-node cluster minimum (2 vCPU, 4 GB RAM per node); 10 GB Persistent Volume for logs; cluster-wide network policies
Runtime Kubernetes v1.26 or later
Configuration deployment.yaml, service.yaml, ingress.yaml, secret.yaml, networkpolicy.yaml
Use Case Enterprise clients requiring orchestration; high-availability demos; multi-tenant environments

4.4 Option D: On-Premises VM / Bare-Metal

Attribute Specification
Components Pre-built binaries; systemd service files; optional local SQLite database
Infrastructure 2 vCPU, 4 GB RAM minimum; 30 GB disk; firewall and VPN access configured
Runtime Ubuntu 22.04 LTS or RHEL 8/9
Configuration systemd.service, config.yaml, firewall rules
Use Case Air-gapped environments; clients with strict data residency requirements; regulated industries

4.5 Option E: Local Development Server

Attribute Specification
Components Source code with npm run dev or python -m http.server
Infrastructure Node.js 20+ or Python 3.10+; 8 GB RAM recommended; localhost only (no public exposure)
Runtime macOS 13+ (Apple Silicon) or Windows 11
Configuration .env.local, dev.env
Use Case Internal development; client demonstrations on consultant's machine; proof-of-concept testing

5. General IT Baseline Requirements

The following requirements apply to all deployment options unless explicitly noted otherwise:

Requirement Description Verification Method
Network Isolation Deploy demos in a dedicated subnet/VPC or isolated Docker network. Only ports 80/443 (or the designated app port) should be accessible externally. nmap scan of public IP; firewall rule audit
TLS Termination All HTTP traffic must use HTTPS with a valid certificate (Let's Encrypt or corporate PKI). TLS 1.2 minimum; TLS 1.3 preferred. curl -I https://demo.example.com returns 200 with trusted certificate
Least-Privilege IAM Service accounts and API tokens must have only the permissions required for operation (e.g., read-only access to static assets, no write access to storage). IAM policy review; aws iam simulate-principal-policy
Secret Management API keys, database passwords, and client-specific tokens must be stored in environment variables or secrets managers (Kubernetes Secrets, AWS Secrets Manager, HashiCorp Vault). Never commit secrets to source code. git grep -i "key\|secret\|password" returns no matches
Patch Management OS and container base images must be updated monthly, or immediately upon release of critical CVE patches (CVSS ≥ 7.0). CI pipeline includes docker scan or trivy; OS runs apt-get update && apt-get upgrade
Logging & Monitoring Capture access logs, error logs, and audit logs. Forward to central SIEM (Elastic, Splunk, or equivalent) with minimum 30-day retention. Verify log shipper (Filebeat, Fluent Bit) is active; sample log query successful
Backup For any persistent state (databases, uploaded files), schedule daily incremental backups stored off-site and encrypted at rest (AES-256). Backup job status visible in cron/scheduler; test restore successful
Disaster Recovery Document and test a rollback procedure (previous Docker image tag, prior git commit, VM snapshot) quarterly. DR test completed in staging environment; results documented

7. Security Development Practices

7.1 Secure Coding Standards

  • Input Validation: Validate all inputs on both client-side and server-side. Reject unexpected data types or formats.
  • Parameterised Queries: Use parameterised queries or prepared statements for all database interactions, including SQLite.
  • Output Encoding: Sanitise any HTML rendered from user data using libraries like DOMPurify (JavaScript) or Bleach (Python).
  • Error Handling: Never expose stack traces or internal error details to end users. Log detailed errors server-side only.

7.2 Static Analysis & Code Quality

  • Run ESLint with eslint-plugin-security for JavaScript/TypeScript projects
  • Run Bandit for Python code security analysis
  • Run Semgrep for cross-language security pattern detection
  • Enforce minimum 80% code coverage for security-critical modules

7.3 Dependency Management

  • Lock all dependency versions in package-lock.json (Node.js) or requirements.txt with pinned versions (Python)
  • Review transitive dependencies quarterly using npm ls or pipdeptree
  • Subscribe to security advisories for critical dependencies

7.4 CI/CD Security Gates

  • Require at least one security-trained reviewer approval for all pull requests
  • Block merges if trivy, snyk, or npm audit reports HIGH or CRITICAL vulnerabilities
  • Run SAST (Static Application Security Testing) on every commit to main/production branches

7.5 Secret Scanning

  • Enable GitHub secret scanning (or equivalent) on all repositories
  • Run git-secrets or trufflehog as a pre-commit hook
  • Rotate any accidentally committed secrets immediately

8. Incident Response Procedure

Note: This procedure applies to security incidents involving demo applications. For production system incidents, refer to the main Incident Response Plan (IRP).
Phase Actions Responsible Party Timeline
1. Detection
  • WAF/SIEM alert for suspicious activity
  • Client report of unexpected behaviour
  • Automated vulnerability scan finding
 Security Operations / Monitoring Immediate
2. Containment
  • Block offending IP addresses via WAF/firewall
  • Halt container deployments or rollouts
  • Isolate affected systems from network
 DevOps / Security Within 1 hour
3. Investigation
  • Pull access, error, and audit logs (last 24-72 hours)
  • Capture container image hash and configuration
  • Identify attack vector and scope of compromise
 Incident Analyst / Security Within 4 hours
4. Eradication
  • Patch vulnerable library or configuration
  • Rebuild container images from clean base
  • Rotate any potentially compromised credentials
 Development / DevOps Within 24 hours
5. Recovery
  • Redeploy patched application
  • Verify functionality with smoke tests
  • Monitor closely for 48 hours
 QA / DevOps Within 48 hours
6. Post-Mortem
  • Document root cause, timeline, and impact
  • Identify improvement actions
  • Update this SOP if necessary
  • Notify affected clients (if applicable)
 All Stakeholders Within 1 week

9. Documentation & Change Management

  • Store this SOP in Git at docs/sop/it-security-sop.md with semantic version tags
  • Any change to deployment topology or security controls must trigger a Change Request in ClickUp/Jira with security team sign-off
  • Review this SOP quarterly or immediately after any security incident
  • Maintain a changelog of all SOP revisions

10. Client-Facing Deliverables

When presenting demo applications to clients, provide the following security documentation:

  1. Architecture Diagram — Showing network isolation zones, public endpoints, data flows, and integration points
  2. Compliance Matrix — Mapping security requirements to implemented controls (aligned to GDPR, ISO 27001, SOC 2 as applicable)
  3. Risk Assessment Summary — Top 5 identified risks with current mitigation status
  4. Data Handling Statement — Clarifying what data is processed, where it's stored, and retention periods
  5. Support & Escalation Contacts — Security incident reporting channel and response SLAs
